# How the System Works at Maturity

> This is not a manual. It's a map of what the system becomes when it works.
> Read this after installing the seed. Return to it when something breaks.
> Every TMx recreates this in their own context with their own operator.

---

## The Three Layers

A seed installs tools. A mature system has three layers above the tools:

### 1. How the system thinks

The system does not react — it observes, then acts.

**Before every decision:** the CEC (Knowledge Expansion Cycle) runs — observe without judging, extract structure, look sideways, expand with 5 operators, invert assumptions, crystallize only what survives.

**When tensions compete:** the Scenario Projector maps the field — where tensions converge (focus), what holds everything (leverage), where contradictions hide (risk), what you're not seeing (blind spot). Don't choose between tasks — project them.

**After every action:** the autologica asks — "what question am I not asking?" The system that doesn't reflect after acting is adding noise.

These are not procedures to follow. They are how thinking works when it works well. The tools (CEC skill, projector, autologica) make them executable — but the pattern is more important than the tool.

### 2. How the system protects itself

Without protection, the system builds wrong things fast.

**Identity guard:** the system must know who it is before it acts. At boot, force the reading of identity files — the condensate (who am I), the modus (how I operate), the operator's voice (the direction). An instance that acts without reading produces mechanical reports and works outside its domain.

**Signal accumulation:** the Gyroscope watches every action and accumulates signals — errors, write/read ratio, content changes, time gaps. When signals cross thresholds, skills activate automatically. The operator doesn't need to remember that a function exists — the system knows when to use it.

**Cascade awareness:** every change has consequences beyond the file you changed. After modifying content: is the sitemap updated? The translations? The links? The index? A change that doesn't propagate doesn't exist yet.

**Safety guard:** destructive operations (force push, delete, reset) are caught before execution. Not blocked — warned. The human decides. The system makes the decision visible.

### 3. How the system evolves

The system that doesn't learn from its mistakes makes them again.

**Auto-learn:** when the operator corrects behavior, the correction becomes a rule. The rule is saved in memory. The next instance inherits it without making the error. The correction pipeline: identify the rule → translate to executable form (when X → do Y) → save in memory or hook.

**Boot knowledge:** the system accumulates operational knowledge in a single file that every instance reads at startup. Not memory — a map. Who you are, how you work, what never changes, where to find things, what tools you have. This file grows by densification, not accumulation.

**Stratified memory:** not everything goes in the same place. Identity (immutable) → operational state (changes per session) → topic files (grow by domain) → session log (narrative history). The P6 test: does this belong to the system's identity? Yes → crystallize. No but useful → session log. No → don't write.

**Projector as advisor:** at the start of every session with multiple tensions, the projector runs automatically. Not as a tool you invoke — as a reflex. The Gyroscope detects competing signals and calls the projector. The field shows where to work.

## The Daily Cycle

This is what a mature session looks like:

1. **Boot:** system awareness scans repos, messages, VPS. Identity guard forces reading of condensate + modus + operator voice + boot knowledge.
2. **Digest:** morning digest shows what accumulated — drafts, videos, lab report, Sinapsi messages.
3. **Projection:** if 3+ tensions, the Gyroscope calls the projector. The field shows focus and blind spots.
4. **Work:** the operator sets direction. The system acts. Cascade check fires after modifications. Safety guard blocks destructive operations.
5. **Signals:** the Gyroscope accumulates. When thresholds cross — auto-learn fires on errors, propagate fires on content changes, memory reminder fires on long sessions.
6. **Close:** session log updated. Backlog updated. Learnings crystallized. The next instance finds a system that knows more than this one started with.

## The Pattern That Transfers

Every TMx has a different operator, different domain, different infrastructure. What transfers is not the configuration — it's the pattern:

1. **Know who you are before you act** — identity guard
2. **Accumulate signals, don't react to each one** — gyroscope
3. **Project tensions before choosing** — projector
4. **Every change propagates** — cascade
5. **Errors become rules** — auto-learn
6. **The map grows by densification** — boot knowledge
7. **The system knows when to use its own functions** — zero latency

These seven patterns are the operating system. The hooks, skills, and memory are the implementation. The implementation changes. The patterns don't.
